# testing-intent-visualization：测试意图 -> 可读结构

这个 skill 用于把测试意图资产展示给人看。目标不是让用户“做题”或阅读全量 YAML，而是让用户按自然顺序理解已有测试意图如何组织。

## 静态页面示例

如果当前项目没有已有 HTML 页面、报告页或前端框架可复用，可以参考或复制 [assets/intent-tree-example.html](assets/intent-tree-example.html) 实现静态页面。

使用要求：

- 示例页只作为结构和交互参考；实际页面要替换为项目真实 `intentions` / `journeys` 数据。
- 保留三栏阅读结构：`业务域`、`流程主干`、`节点详情`。
- 保留“只看流程 / 展开到覆盖点”的层级控制，避免默认全量展开。
- 保留“可能漏掉的意图”独立面板，不要把 `edge_branches` 混入已有测试意图树。
- 没有构建系统时，交付一个可直接打开的自包含 HTML；有现成页面时，优先按项目技术栈复用组件和样式。

## 核心原则

1. **先展示已有结构，再谈缺口。**
   已有测试意图树和“可能漏掉的意图识别”必须分开。不要把 `edge_branches`、review item 或 gap 混进已有意图树里。

2. **按人的阅读顺序组织。**
   推荐顺序是：

   ```text
   业务域 -> 流程 -> 覆盖点 -> 相关测试说明 -> 检查点 / 边界情况
   ```

3. **默认只展示主干。**
   不要默认展开几百个节点。默认让用户先看到业务域和流程；覆盖点、测试说明、检查点按需展开或在详情区查看。

4. **同一概念只用一个名称。**
   使用：
   - `业务域`
   - `流程`
   - `覆盖点`
   - `相关测试说明`
   - `独立测试说明`
   - `检查点`
   - `边界情况`

   避免在用户界面里混用 `branch`、`journey`、`intent`、`gap`、`matrix`、`asset` 等内部词。

## 推荐页面结构

优先使用三栏阅读结构：

```text
左栏：业务域
中栏：当前业务域的流程主干树
右栏：选中节点详情
```

### 左栏：业务域

每个业务域卡片显示：

- 业务域名称，例如 `工单`、`漏洞`、`登录/用户`
- 流程数量
- 测试说明数量
- P0 数量

用户先选业务域，不要一开始展示全局大树。

### 中栏：流程主干树

当前业务域内展示：

```text
业务域
  流程
    已覆盖点
      覆盖点
        相关测试说明
    相关测试说明
      测试说明
  独立测试说明
    测试说明
```

说明：

- `已覆盖点` 来自 journey 的 `branches`。
- `相关测试说明` 来自 branch 引用的 intent。
- `独立测试说明` 是已经存在但没有被该业务域任何流程引用的 intent。它不是错误，也不是缺口。
- 不要把 `edge_branches` 放进这棵树。

### 右栏：节点详情

按选中节点类型显示：

- 业务域：流程列表、高优先级测试说明、独立测试说明摘要。
- 流程：业务上下文、覆盖点数、相关说明数、覆盖点表格。
- 覆盖点：触发条件、预期结果、相关测试说明。
- 测试说明：业务上下文、检查点、边界情况、绑定的测试文件。

检查点和边界情况通常不要放进左侧树作为节点；它们数量大，适合放在详情区。

## 数据映射

### Intent

单条 intent 展示为 `测试说明`。

字段映射：

- `title` -> 测试说明标题
- `business_context` / `precondition` -> 详情说明
- `assertions` -> 检查点
- `edge_cases` -> 边界情况
- `specBindings` -> 绑定的测试文件
- `priority` / `status` -> 标签或统计

### Journey

单条 journey 展示为 `流程`。

字段映射：

- `title` -> 流程标题
- `business_context` -> 流程说明
- `branches` -> 已覆盖点
- `branches[].condition` -> 覆盖点的“什么时候发生”
- `branches[].result` -> 覆盖点的“应该发生什么”
- `branches[].intent` -> 相关测试说明

### Edge Branch

`edge_branches` 表示“可能漏掉的意图识别”，不是已有意图树的一部分。

如果用户明确要求展示缺口，单独做一个视图或面板，例如：

```text
可能漏掉的意图
  业务域
    所属流程
      条件 / 期望结果 / 原因
```

不要把它混在 `已覆盖点` 下。

## 命名规范

推荐中文 UI 文案：

- `已有测试意图树`
- `业务域`
- `流程主干`
- `已覆盖点`
- `相关测试说明`
- `独立测试说明`
- `检查点`
- `边界情况`
- `绑定的测试文件`

避免：

- `未挂到流程的测试说明`，改用 `独立测试说明`
- `流程里已经写到的情况`，改用 `已覆盖点`
- `关联的测试说明`，改用 `相关测试说明`
- `判断题队列`，除非用户明确要求人工确认界面
- `资产化`、`基线`、`gap`、`edge branch` 等内部词

## 交互规则

- 默认选中第一个业务域。
- 默认只展开到流程层级。
- 提供 `展开到覆盖点` 和 `只看流程` 两个操作。
- 搜索应限定在当前业务域内，避免全局结果淹没用户。
- 搜索左侧树时，只搜索可见层级和测试说明标题；检查点可作为测试说明详情展示，不必直接把所有检查点变成搜索结果节点。
- 点击节点时，右侧详情必须更新，不要跳转离开当前阅读上下文。

## 质量检查

交付前检查：

1. 页面默认不是全量展开。
2. 用户能在首屏看到业务域列表和当前业务域流程主干。
3. 树里没有 `edge_branches` / 可能漏掉的意图。
4. `独立测试说明` 只表示“已有但未被流程引用”，不能暗示错误。
5. 右侧详情能解释选中节点的上下文。
6. 检查点和边界情况在测试说明详情中可见。
7. 搜索、展开/收起、业务域切换可用。
